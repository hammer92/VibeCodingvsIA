Implementación de MCP

El Problema

Necesitas que la IA lea tu base de datos local para entender la estructura de las tablas antes de escribir una query SQL.

La Solución (Código)

Configuras un Servidor MCP en el archivo de configuración del IDE. Esto crea un puente seguro.

  • La IA detecta automáticamente la herramienta query_db.
  • No envías credenciales a la nube (todo corre local via Docker).
config_mcp.json
// Configuración real para conectar Cursor a Postgres
{
  "mcpServers": {
    "local-db": {
      "command": "docker",
      "args": [
        "run", "-i", "--rm",
        "-e", "POSTGRES_CONNECTION_STRING=postgresql://user:pass@host/db",
        "mcp/postgres-server"
      ]
    },
    "github-tools": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"]
    }
  }
}

Implementación de Spects

El Problema

La IA suele olvidar qué librerías usas o escribe código con estilos mezclados (CSS vs Tailwind).

La Solución (Código)

El archivo .cursorrules actúa como un Senior Engineer que revisa cada PR.

Antes de generar cualquier respuesta, el modelo lee este archivo. Si le pides "Crea un botón", automágicamente lo hará en Tailwind y creará el archivo de test, porque la regla lo exige.

.cursorrules
# ESTE ARCHIVO GOBIERNA EL COMPORTAMIENTO DE LA IA

Tech Stack:
- Framework: Next.js 14 (App Router)
- Styling: Tailwind CSS
- State: Zustand

Behavior Rules:
1. ALWAYS create a corresponding `.test.tsx` file when creating component.
2. NEVER use `console.log` for debugging, use custom `Logger`.
3. If modifying API routes, update `openapi.yaml` automatically.

Tone:
- Be concise. Do not explain the code unless asked.
Amazon Q Developer

Implementación de Workflows (/dev)

El Concepto

En lugar de chat, usas el comando /dev para iniciar un Agente de Planificación.

La Implementación

Amazon Q no solo escribe código, crea un plan de ejecución (Workflow) que puedes revisar antes de aplicar.

OUTPUT DE CONSOLA SIMULADO
> /dev add login feature [PLANNING] Analyzing 14 files... [WORKFLOW GENERATED] 1. Create /src/auth/Login.tsx 2. Update /src/App.tsx router 3. Run 'npm test' to verify Confirm implementation? (Y/n)
q-agent-spec.yaml
# Definición interna del Workflow (Conceptual)

agent_task: "Implement Feature"
steps:
  - read_context: ["src/components", "package.json"]
  - plan_changes:
      output_format: "markdown_list"
  - generate_code:
      sandbox_execution: true
  - verify:
      command: "npm run test:unit"
      on_failure: "auto_fix_and_retry(max=3)"
Kiro Specs

Specs: Desarrollo Incremental

El Concepto

Los Specs son documentos markdown que formalizan el diseño e implementación de features complejas. Permiten iterar en requerimientos antes de ejecutar.

La Implementación

Creas un archivo en .kiro/specs/ con la estructura de tu feature. El agente lo analiza, genera un plan de tareas y lo ejecuta incrementalmente.

Los specs soportan referencias a archivos externos usando #[[file:openapi.yaml]], ideal para integrar especificaciones de API o GraphQL en el proceso de desarrollo.

.kiro/specs/auth-feature.md
# Spec: Sistema de Autenticación

## Objetivo
Implementar autenticación JWT con login y registro.

## Contexto
- Framework: Next.js 14
- DB: PostgreSQL + Prisma
- API: #[[file:api/openapi.yaml]]

## Requerimientos
1. POST /api/auth/login
2. POST /api/auth/register
3. Middleware de autenticación
4. Validación con Zod

## Criterios de Aceptación
- [ ] Tests con cobertura > 80%
- [ ] Rate limiting en endpoints
- [ ] Logs de intentos fallidos
Google Antigravity Deepmind

Orquestación Multi-Agente

El Enjambre (Swarm)

Antigravity no trabaja solo. Actúa como un Orquestador Principal que instancia y coordina sub-agentes especializados para tareas complejas en paralelo.

La Configuración (YAML)

Defines la topología del equipo en swarm.yaml. Aquí asignas roles, herramientas específicas y permisos a cada sub-agente.

El orquestador divide la tarea del usuario (ej: "Crear una landing page") y delega: el Designer genera assets, el Coder implementa y el QA valida.

.antigravity/swarm.yaml
# Definición de Roles para Ingeniería Compleja
swarm_topology:
  orchestrator: "Antigravity Core 2.0"

  specialists:
    - id: "ui_designer_agent"
      tools: ["generate_image", "css_inspector"]
      objective: "Ensure visual aesthetics"

    - id: "qa_guardian"
      tools: ["browser_automation", "jest_runner"]
      policy: "Block merge if coverage < 90%"

  workflow_logic:
    parallel_execution: true
    shared_context: ["memory_bank", "file_system"]
Standard Protocol

El Estándar agent.md

Definición Unificada

El archivo agent.md se ha convertido en el manifiesto universal para definir la identidad, herramientas y permisos de un agente, independiente del IDE.

Soporte Multi-IDE

Este estándar permite portabilidad total entre entornos. Define una vez, ejecuta donde sea.
